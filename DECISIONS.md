# DECISIONS.md — Architecture Decisions

## 1. Architecture Decisions

### UUID vs BIGSERIAL

**Decision: UUID everywhere.**

Reasoning:
- In a distributed microservices system, IDs are generated by multiple services independently. UUID (generated with `gen_random_uuid()`) requires no coordination — each service generates unique IDs without a central sequence.
- BIGSERIAL requires a single DB sequence per table, which creates coupling and doesn't work cleanly when services have separate databases.
- Trade-off: UUID v4 indexes are non-sequential, leading to more B-tree page splits than BIGSERIAL on high-volume inserts. At 100M+ rows this matters — solution would be UUID v7 (time-ordered) or ULID.

### NUMERIC vs DOUBLE for Money

**Decision: `NUMERIC(19, 2)` for all monetary values.**

Never use `FLOAT` or `DOUBLE` for money. IEEE 754 floating point cannot represent 0.10 exactly:
```java
System.out.println(0.1 + 0.2); // → 0.30000000000000004
```
`NUMERIC` in PostgreSQL stores exact decimal values. Java `BigDecimal` maps perfectly.

### Why TIMESTAMPTZ not TIMESTAMP

`TIMESTAMP` stores wall-clock time with no timezone info. When the server changes timezone or a user is in a different TZ, the meaning of the value changes.

`TIMESTAMPTZ` stores UTC internally and converts to the session timezone on retrieval. This is always correct regardless of where the code runs. All temporal columns use `TIMESTAMPTZ`.

## 2. Kafka Design

### What Kafka Key to Use?

**Decision: `rentalId` as Kafka key for all rental-flow topics.**

Kafka partitioning: all messages with the same key go to the same partition, which means they are consumed in order.

For a single rental flow (`acquire-cabinet-lock-event`, `payment-request`, `eject-powerbank-event`), ordering matters: payment must come after lock, eject must come after payment. Using `rentalId` as key ensures all events for the same rental land on the same partition and are processed in order.

For `payment-events`, we use `paymentId` as key — ordering per payment matters (PENDING → SUCCESS sequence must not be reversed).

### What Happens if Kafka is Unavailable?

Current implementation: fire-and-forget with error logging. If Kafka is down:
- The event is lost.
- The rental FSM gets stuck in intermediate state (ACQUIRING_LOCK, PAYMENT_PROCESSING, EJECTING).
- User sees polling returning the same intermediate state indefinitely.

**Production solution: Outbox Pattern**
1. Write event to `outbox` table in the same DB transaction as the state change.
2. A background poller reads the outbox table and publishes to Kafka.
3. On success, mark the outbox record as `published = true`.
4. This guarantees at-least-once delivery even when Kafka is down.

### DB + Kafka Transactionality

**Problem:** Record saved to DB → Kafka publish fails → inconsistency.

Without Outbox: we have exactly this problem. The DB state says PAYMENT_PROCESSING but the payment-service never received the event.

**Production solution: Transactional Outbox Pattern**
```sql
BEGIN;
  UPDATE rentals SET status = 'PAYMENT_PROCESSING' WHERE id = :id;
  INSERT INTO outbox (topic, key, payload) VALUES ('payment-request', :key, :payload);
COMMIT;
```
The outbox poller then publishes from the outbox with retry, giving guaranteed delivery without XA transactions.

## 3. Idempotency Key: Conflict Handling

**Decision:** When `idempotency_key` already exists in DB, return the **existing record with HTTP 200**.

Do NOT return 201 (Created) because nothing was created. The semantics: "you already did this, here's what you got."

**Same key + different amount:**
We reject the difference silently — we return the original payment. The rationale: the idempotency key contract says "same key = same request." If the amount is different, the client has a bug. We should NOT process a new payment because the client's retry protection would be broken.

**Race condition handling:**
If two threads receive the same idempotency key simultaneously, one will hit a `DataIntegrityViolationException` on the UNIQUE constraint. We catch this exception and do a fallback read to return the existing record.
